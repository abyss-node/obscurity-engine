# 10x Roadmap — 2026-07-02

A feature roadmap derived from a full codebase review (backend, frontend,
eval harness, docs). Companion to [roadmap.md](roadmap.md) — that file records
what's been tried and closed; this one is the forward plan for making the
product an order of magnitude more valuable.

## The thesis: three ceilings

Today the product is a **stateless, anonymous, one-shot novelty tool**: type a
Last.fm username → obscurity index + up to 25 obscure artist recs → share a
PNG → leave. Nothing changes on the next visit. The review found three
structural ceilings, and every 10x feature attacks one of them:

1. **Data ceiling.** Candidates come exclusively from Last.fm `artist.getsimilar`,
   which is popularity-biased and genre-clustering. Proven at n=348: reach is
   flat (~0.05) across every scoring lever; pure-discovery hits are pinned at
   ~78 no matter what. No ranking change fixes this — only a richer graph (CEGE).
2. **Signal ceiling.** The engine emits recommendations and never learns
   anything back. No clicks, saves, dismissals, or follows are captured. The
   eval's ground truth is *unattributed* holdout replays — a hit counts whether
   the engine caused it or the user found the artist elsewhere
   (`eval/metrics.py:5-8` admits the confound). We cannot measure, let alone
   optimize, the thing the mission cares about: **did we cause a discovery?**
3. **Audience ceiling.** Requires an existing Last.fm account with scrobble
   history — a small niche. No accounts, no server-side user state (identity is
   a query param; persistence is localStorage), so there is also no retention
   loop for the users we do get.

Corollary from the mission ("help under-discovered artists reach their true
1000 fans"): the product currently serves only the *listener* half. The artist
half — the actual beneficiary — has no surface at all.

**Dormant assets the review found** (cheap wins, already built):
- Full track pipeline live at `/api/discovery/tracks` (ungated server-side,
  hidden client-side behind `TracksComingSoon`).
- Complete Spotify PKCE OAuth + playlist-creation flow in `frontend/src/lib/spotify.ts`
  — dead code today because only tracks mode reaches it.
- `velocity: Option<f64>` field stubbed on every response item (`models.rs:48`,
  always `None`) — Last.fm exposes no listener time-series, but *we fetch
  `artist.getinfo` listener counts for ~600 artists per request and throw the
  numbers away*. Persisting them builds the missing momentum dataset ourselves.
- Persistent share links + Redis-ready cache landing tonight (see
  [dev-tasks-overnight-2026-07-02.md](dev-tasks-overnight-2026-07-02.md)).

## Phase 1 — Close the loop (signal ceiling → data flywheel)

The highest-leverage phase. It converts the app from a demo into a system that
generates the exact data (attributed adoptions) that the eval harness proved it
lacks, and creates the first retention loop.

### F1. Server-side persistence layer (Postgres)

**What:** Add the first database. Postgres (Railway addon), accessed from the
Rust backend via `sqlx`. Tables: `users` (id, lastfm_username, created_at,
settings), `runs` (user, period, appetite, depth_score, top_genres, timestamp),
`recommendations` (run_id, artist, rank, conviction, listeners, scores),
`events` (rec_id, type, timestamp), `artist_observations` (artist,
listeners, monthly_listeners, observed_at) — see F4.

**Why:** every other Phase-1/2 feature needs it; today literally nothing
survives a backend restart except contributed API keys.

**Acceptance criteria:**
- `DATABASE_URL` unset → backend boots and serves `/api/discovery` exactly as
  today (graceful fallback, verified by integration test with the var absent).
- With `DATABASE_URL`: every discovery run inserts one `runs` row + N
  `recommendations` rows; verified by a test that runs a discovery and asserts
  row counts and score fields match the HTTP response.
- Migrations are versioned (`sqlx migrate`) and run automatically on boot;
  restarting the service loses zero rows.
- p50 latency overhead of persistence on `/api/discovery` < 50ms (measured
  before/after on a warm cache hit).

### F2. First-party feedback capture

**What:** Instrument every recommendation surface: listen-link clicks
(Spotify/Last.fm/Bandcamp/This-Is), an explicit **save** (bookmark this
artist) and **dismiss** (not for me) on each `ArtistCard`, and share events.
New endpoint `POST /api/events` (rec_id, event_type); frontend fires it
best-effort (never blocks UI). Saved artists get a "Saved" view.

**Why:** this is the attributed-adoption signal the eval fundamentally lacks.
Dismissals are equally valuable (explicit negatives don't exist anywhere in the
current data).

**Acceptance criteria:**
- Clicking any listen link or save/dismiss writes exactly one `events` row with
  the correct rec_id and type (integration test with a stubbed run).
- Events endpoint rejects malformed/unknown rec_ids (400) and is rate-limited
  per IP; verified by test.
- UI: save/dismiss state survives page reload (server-backed, not localStorage);
  dismissed artists never reappear in future runs for that user (scoring
  exclusion list, verified by two consecutive runs in a test).
- Zero UI regression when backend has no DB: buttons hide or no-op gracefully.

### F3. Closed-loop adoption tracking ("Your discoveries")

**What:** A scheduled job (tokio cron in-backend) re-polls each active user's
recent scrobbles weekly and marks a recommendation **adopted** when the user
scrobbled that artist ≥3 times *after* the rec was shown (mirrors the eval's
`min_future_plays=3` definition, but now attributed and per-rec). Surface it:
a "Your discoveries" page — "You've adopted 7 artists the engine showed you,
mean listeners 9.4K" — plus obscurity-index-over-time from `runs`.

**Why:** this is the product's proof-of-value made visible to the user (the
retention hook), and it produces a **first-party attributed eval metric** —
adopted-per-shown — that the n=348 harness cannot compute. Because users are
scrobblers, attribution comes free: rec shown at T, scrobbles after T are
observable.

**Acceptance criteria:**
- Job correctness: unit test with fixture scrobble data — artist recommended at
  T and scrobbled 3+ times in (T, T+7d] → marked adopted; scrobbled before T or
  <3 times → not.
- A user with ≥2 runs sees an index-over-time sparkline and a discoveries list;
  verified in headless-browser QA.
- New eval capability: `eval/` gains a `first_party_report.py` that reads the
  events/adoptions tables and prints adopted-per-shown, save-rate, dismiss-rate
  with bootstrap CIs — the online counterpart of obscW@k. Runs against staging
  data without touching the existing harness.
- Privacy: per-user data export + delete endpoint (`GET/DELETE /api/me/data`),
  tested.

### F4. Listener-count time series → real velocity signal

**What:** Persist every `artist.getinfo` observation the pipeline already makes
(artist, listeners, monthly listeners, timestamp) into `artist_observations`
(dedup: one row per artist per day). After ~60 days of accumulation, compute
listener growth rate and populate the stubbed `velocity` field; show a
"rising" badge on artists with high relative growth.

**Why:** Last.fm has no listener time-series API — this dataset cannot be
bought, only accumulated, and the backend already fetches ~600 getinfo calls
per discovery run and discards them. It's also the seed of the artist-side
product (F9): "your listener growth curve."

**Acceptance criteria:**
- Observations written as a side effect of normal discovery runs with zero
  added Last.fm calls (verified: request count unchanged in a counting test).
- Dedup: same artist observed twice in one day → one row (test).
- After backfill window, `velocity` is non-null for artists with ≥2
  observations ≥14 days apart; formula unit-tested on fixtures.
- The badge appears only above a growth threshold; QA screenshot check.

## Phase 2 — Widen the front door (audience ceiling)

### F5. Spotify listening import (no Last.fm required)

**What:** "Continue with Spotify" on the landing page: user OAuth (the PKCE
plumbing in `lib/spotify.ts` already does the hard part), pull top artists
(short/medium/long term) + recently played, map them to the seed format
(`log2`-weighted), and run the same pipeline. Backend gets a
`POST /api/discovery/spotify` accepting the seed payload (token stays
client-side; backend never sees it).

**Why:** the single biggest TAM unlock — Spotify's user base vs Last.fm's is
orders of magnitude. The engine's only hard dependency on Last.fm for *the
user* is seed construction; candidates/scoring are user-agnostic.

**Acceptance criteria:**
- A Spotify account with zero Last.fm history gets a full results page (index,
  matrix, ledger) — end-to-end QA with a real test account.
- Seed parity test: for a user with both accounts, Spotify-derived and
  Last.fm-derived seed sets overlap ≥60% on top-20 artists (sanity fixture,
  not a hard product gate — documents the mapping quality).
- Spotify tokens never appear in backend logs or storage (grep test on debug
  logs during an e2e run).
- Time-range picker maps to the existing period pills; blend mode works by
  weighting the three Spotify ranges (unit test on weights).
- Without `NEXT_PUBLIC_SPOTIFY_CLIENT_ID`, the button is hidden and Last.fm
  flow is unchanged.

### F6. One-click playlist export for artist recs

**What:** "Export as playlist" on the results page: top 2 tracks per
recommended artist (via existing `artist.gettoptracks`) → private Spotify
playlist named "Obscurity finds — {date}", using the already-built
`createPlaylist`. Each export is an F2 event.

**Why:** recommendations you can *press play on* get adopted; links you have to
chase don't. Also makes every adopted rec attributable (playlist plays →
scrobbles → F3 closes the loop). Currently the entire playlist code path is
dead.

**Acceptance criteria:**
- E2E: export from a real results page creates a playlist with ≥80% of shown
  artists represented (some artists lack Spotify presence — that's expected;
  the export reports "18 of 25 added").
- Export without a connected Spotify session triggers OAuth then resumes and
  completes (session-storage handoff already exists — verify it survives).
- Playlist is private by default; an `events` row of type `playlist_export` is
  written per export (test).
- Failure of any single track lookup never aborts the export (unit test with a
  stubbed 404).

### F7. Public taste pages + shared-result upgrade (SEO / acquisition)

**What:** Upgrade tonight's persistent share links (`/r/[id]`) into permanent
public taste profiles (`/u/[username]`, opt-in): obscurity index, taste
signature, top discoveries, with proper OG tags and server rendering so links
unfurl on WhatsApp/Twitter/Discord. Add programmatic artist pages
(`/artist/[name]`) for every artist that has ever appeared in
recommendations: listeners, genre, "listeners with taste like X adopt this
artist", listen links.

**Why:** the only acquisition channel today is a PNG. Public pages make every
share a landing page and every obscure artist an SEO surface — thousands of
long-tail pages for artists nobody else writes about. Directly serves the
mission (artist visibility) before CEGE lands.

**Acceptance criteria:**
- `/u/[username]` 404s unless the user opted in (default off); opt-out deletes
  the page (test both).
- Link unfurl: OG title/description/image present in server-rendered HTML
  (curl test, no JS execution).
- Artist pages render from `artist_observations` + `recommendations` data with
  no live Last.fm call on the request path (latency test: p50 < 300ms).
- Lighthouse SEO score ≥ 90 on both page types; sitemap.xml lists them.

## Phase 3 — Break the reach ceiling (data ceiling)

### F8. Pluggable candidate provider + CEGE integration

**What:** Extract a `CandidateSource` seam where `candidates.rs` calls
`getsimilar` (and its mirror in `eval/pipeline.py`), then implement a CEGE
provider: seed artists → CEGE graph neighbors (pgvector similarity + graph
edges from the MusicBrainz/Wikidata ETL). Ship behind a flag; blend mode
(union of Last.fm + CEGE candidates) is the expected winner. The eval harness
A/Bs it exactly like every prior lever — this is why the scoring brain was
kept Python-portable.

**Why:** the single documented unlock. Every closed experiment in
[roadmap.md](roadmap.md) concluded reach is capped by the candidate source,
not the ranking. This is the only feature on this list that can move
pure-discovery hits off the ~78 ceiling.

**Acceptance criteria (the bar is pre-registered, per eval discipline):**
- Eval harness runs the CEGE provider on the de-biased cohort (n=348 protocol:
  126 users × 3 anchors, paired bootstrap) with zero changes to metrics code.
- **Ship gate:** reach CI excludes the 0.05 baseline upward, AND
  pure-discovery hits exceed 78 with a paired-bootstrap CI excluding zero, AND
  obscW@k does not regress. If CEGE candidates don't clear this, the feature
  does not ship (same rule that killed two-hop).
- Serving latency with blend ≤ 1.5× Last.fm-only p50 (candidate fetch is
  parallel; CEGE is a local DB).
- Provider behind `CANDIDATE_SOURCE=lastfm|cege|blend` env; `lastfm` remains
  the default until the ship gate passes.

### F9. Tracks mode, un-gated

**What:** Bring the track pipeline to artist-pipeline standards — fail-closed
error handling (it currently silently drops failed stages and can serve
non-deterministic partials), appetite/underexplored support, populated
`source_seeds` provenance, cross-validation — then remove `TracksComingSoon`.
Sequenced after F8 because the 2026-06-21 deferral reason (same capped graph)
dissolves once candidates come from CEGE; the playlist-export flow (F6) makes
tracks the natural unit.

**Acceptance criteria:**
- Fail-closed parity: injected transient failure in any stage fails the whole
  request rather than returning partials (test per stage — this currently
  fails by design of the old code).
- `source_seeds` non-empty for every returned track (test).
- Appetite slider affects tracks mode identically to artists (e2e).
- Eval: a `track_eval` smoke run on ≥20 users completes and reports the same
  metric family; obscW@k for tracks published as the new baseline (no ship
  gate vs artists — it's a new surface — but the number must exist).

## Phase 4 — The artist side (the mission's other half)

### F10. Artist dashboard ("path to 1000 true fans")

**What:** An artist (or manager) claims their page from F7 (verification:
edit the artist's Last.fm wiki or link from their official site/Bandcamp —
pragmatic v1). Dashboard: listener growth curve (F4 data), distance-to-1000
true-fans estimate (the `true_fans` model from `eval/config.py` finally has a
purpose — as an *artist-facing* metric it was never wrong, it was wrong as a
listener-side ranking gate), where they surface in recommendations (how many
times shown/clicked/saved/adopted, from F1-F3 data), and which genres/taste
clusters adopt them.

**Why:** this is the mission. Every prior phase generates the data that makes
this dashboard non-trivial: observations (F4) give the growth curve,
events (F2) give shown→adopted funnels, CEGE (F8) gives graph context. It's
also the seed of a B2B/artist-services revenue path that the listener app
alone never has.

**Acceptance criteria:**
- An unclaimed artist page shows a "claim this page" flow; claim requires
  passing one of the two verification checks (both e2e-tested with a test
  artist).
- Dashboard renders shown/clicked/saved/adopted counts that exactly match a
  SQL ground-truth query on the events tables (reconciliation test).
- True-fans estimate is computed with the existing `eval/config.py` per-user
  model parameters, unit-tested against 3 hand-computed fixtures.
- No listener-side ranking change of any kind ships with this feature (the
  eval already ruled; diff-scope test asserts `scoring.rs` untouched).

## Sequencing and dependencies

```
Tonight (in flight): share persistence, Redis cache, test harness, polish
Phase 1: F1 → F2 → F3 (strict chain), F4 (parallel after F1)
Phase 2: F5, F6, F7 (independent of each other; all need F1; F6 needs F5's OAuth)
Phase 3: F8 (needs CEGE ETL from the CEGE project), F9 (needs F8)
Phase 4: F10 (needs F2/F3/F4 accumulated data — start ~60 days after Phase 1 ships)
```

Phase 1 is deliberately first even though Phase 2 is flashier: every week
without feedback capture is a week of adoption data lost forever, and F4's
time-series only becomes valuable with age.

## Explicitly not on this roadmap

- Any listener-side scoring/ranking lever (two-hop, match-weight, velocity
  ranking tilt, temporal weighting, adaptive backfill, genre-relative or
  true-fans *ceilings*) — all closed by the n=348 re-validation. F8 changes
  the *candidate pool*, which is the one thing the eval said is the bottleneck.
- Social features (leaderboards, friend compare) — fun, but they don't attack
  a ceiling; revisit after Phase 2 ships and retention is measurable.
  *(Challenged in review — friend-compare is the proven growth loop in this
  category (Obscurify/Receiptify); surfaced as taste decision at the gate.)*
- Native apps — the mobile web layout already works; no evidence of need.
