## What shipped

### T1 ‚Äî Persistent share links
`POST /api/share` + `GET /api/share/{id}` + a server-rendered `GET /r/{id}`
page. Backed by Vercel KV when `KV_REST_API_URL` + `KV_REST_API_TOKEN` are
set, else an in-process in-memory `Map` (zero-config, ephemeral) ‚Äî no
behavior change if you do nothing. 30-day TTL, ~100KB payload cap, strict
payload validation, byte-identical roundtrip. Documented in
[reference-api.md](reference-api.md).

in-memory by default (unchanged behavior), Redis-backed when `REDIS_URL` is
set. Unreachable/misconfigured Redis degrades to a cache miss ‚Äî never fails a
request. Documented in [howto-deploy.md](howto-deploy.md).

| `cd frontend && npm run build` | ‚ùå **fails ‚Äî pre-existing environment issue, not a regression** (see below) |
| `cd frontend && npm test` | ‚úÖ green ‚Äî 4 files / 27 tests passed (`vitest run`) |

**`npm run build` failure ‚Äî root cause and scope:** `next-pwa`'s
workbox-webpack step generates a service-worker bundle whose import
statements are absolute file paths. This machine's home directory contains a
literal apostrophe (`C:\Users\Arnuv'\...`), and the generated `sw.js` treats
that apostrophe as a string terminator, producing invalid JS
(`Missing semicolon`). **This is reproducible on an unmodified `main` at the
exact same path** ‚Äî verified before merging any of T1‚ÄìT4's changes ‚Äî so it
predates and is unrelated to this build. It is a **local Windows dev-machine
path quirk only**: Vercel's build runners use a Linux path with no apostrophe,
so **production deploys are unaffected**. `next dev` is also unaffected
(`next-pwa` disables itself outside `NODE_ENV=production`), which is how the
e2e smoke test below was run. Fixing it (e.g. `outputFileTracingRoot`,
disabling the PWA plugin locally, or moving the repo off an apostrophe'd
path) touches `frontend/next.config.js`, which is code ‚Äî out of scope for
this docs-only task. Recommend a follow-up task if a local production build
is ever needed on this machine.

## End-to-end smoke test (Step 2)

Backend run on `PORT=8099` (default 8080 was occupied by an unrelated local
service on this machine) with the real `LASTFM_API_KEY`/`LASTFM_API_KEYS`
from `backend/.env`; frontend run via `npm run dev -p 3099` with
`NEXT_PUBLIC_BACKEND_URL=http://localhost:8099`.

1. **Real compute for a known Last.fm user** (`rj`, `period=6month`):
   `GET /api/discovery?username=rj&period=6month` ‚Üí `200`, 18 artists, first
   call took **36.2s** (cold ‚Äî server log: `Cache miss: reverse_scrobble:rj:6month:balanced`).
2. **T2 cache hit on identical second run:** same request again ‚Üí `200` in
   **0.28s** (~130√ó faster), server log: `Cache hit: reverse_scrobble:rj:6month:balanced`.
   (Aside: the cached response differs from the first by float ULP noise ‚Äî
   e.g. `15.468298755186723` vs `...725` ‚Äî on JSON round-trip through the
   cache; functionally identical, not a correctness issue, noted for
   awareness.)
3. **T1 share flow, fresh context:**
   - `POST /api/share` with the computed payload (`mode: "artists"`) ‚Üí `201 { "id": "uNWt9S1bcv" }`.
   - `GET /api/share/uNWt9S1bcv` ‚Üí `200`, byte-identical to the POSTed body (`diff` clean).
   - `GET /r/uNWt9S1bcv` (plain `curl`, no cookies/localStorage ‚Äî i.e. a fresh,
     no-storage client) ‚Üí `200`, server-rendered HTML containing the real
     recommendation names (`Alain Boublil`, `Days N' Daze`, ‚Ä¶). Confirms the
     share renders correctly for a brand-new visitor, not just the browser
     that created it.
   - Sanity check: an earlier POST with `mode: "balanced"` (wrong value)
     correctly rendered the "link expired" state at `/r/{id}` ‚Äî the mode
     validation path works as designed.

All local processes (backend, frontend dev server) were stopped after the
smoke test; nothing was left running.

## The two env-var setups you need to do to activate this in prod

### 1. Redis for the backend cache (Railway)
1. Provision Redis (Railway's Redis plugin, or Upstash, etc.) and copy the
   connection string.
2. Railway dashboard ‚Üí backend service ‚Üí set `REDIS_URL` = that string.
3. `railway up -d -y` (or let auto-deploy pick it up once wired). Check
   `railway logs` for `Cache store: Redis (REDIS_URL set, connected)`.
4. **Skip this and nothing breaks** ‚Äî unset `REDIS_URL` keeps today's
   in-memory-cache behavior exactly.

### 2. Vercel KV for persistent share links (frontend)
1. Vercel dashboard ‚Üí project ‚Üí **Storage** ‚Üí **Create Database** ‚Üí **KV**.
   Connect it to the project.
2. Vercel auto-populates `KV_REST_API_URL` + `KV_REST_API_TOKEN` on the
   project env vars ‚Äî confirm under **Settings ‚Üí Environment Variables** (or
   `vercel env ls`).
3. Redeploy. Share links created after this point persist across redeploys
   and are shared across serverless instances (30-day TTL either way).
4. **Skip this and nothing breaks** ‚Äî share links still work, just
   per-instance/ephemeral (in-memory), same as today's non-persistent
   behavior. Only one of the pair set is treated as "not configured" too
   (both are required together).

Full details for both: [howto-deploy.md](howto-deploy.md).
