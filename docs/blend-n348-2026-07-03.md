# Blend candidate-source re-validation at full power (n=348)

**Date:** 2026-07-03
**Verdict:** **SHIP** — all three pre-registered gate conditions met.
Pooled reach +0.0066 absolute (+9.5% relative), 95% CI [+0.0039, +0.0100],
excludes zero; obscurity-weighted@k and pure-discovery hits show no
significant regression. Reach is individually significant in **all three
anchors**, not just pooled.

---

## Question

Spike T-C (see `docs/spike-listenbrainz-2026-07-03.md`) found `blend`
(Last.fm + ListenBrainz candidates merged) beat the shipped Last.fm-only
baseline on reach at n=54, single anchor — a directional GO only. This doc
is the confirmatory run at full power: the 126-user de-biased cohort ×
3 anchors = 348 (user, anchor) samples, the same protocol that decided the
mult-2 ladder (`ab0911e`, "high-power re-validation n=348").

## Pre-registered ship gate (fixed before seeing numbers)

`blend` ships as candidate source **only if**:

1. pooled reach paired-difference CI excludes 0 **upward**, AND
2. obscurity-weighted@k CI does **not** exclude 0 downward (no significant
   regression), AND
3. pure-discovery hits are not significantly regressed.

Anything else = NO-SHIP, numbers reported honestly either way.

## Method

- **Cohort / anchors:** `eval/cohort_debiased.txt` (126 users), anchors
  2026-06-10, 2025-12-10, 2025-06-10. Split: seeds = [-365d], holdout =
  [180d]. Users with no eligible holdout at an anchor drop out of that
  anchor's sample (114/116/118 eligible → 348 samples).
- **Blend arm:** run per-anchor (to checkpoint against the ~1 req/s
  ListenBrainz/MusicBrainz rate limit), each invocation:

  ```
  python harness.py --cohort cohort_debiased.txt --anchor <DATE> \
    --threshold flat --novelty-model underexplored --underexplored-mult 2.0 \
    --xval-genre-overlap --candidate-source blend \
    --json blend_n348_anchor_<DATE>.json
  ```

  then merged with `eval/merge_anchor_runs.py` into `eval/blend_n348.json`
  (348 samples, same shape as a native `--anchors a,b,c` artifact).
- **Baseline:** `eval/vs_mult2.json` — the existing 126-user × 3-anchor
  lastfm-arm run (same cohort, same anchors, same config apart from
  `candidate_source`). It predates the `candidate_source` field, but T-C's
  default-safety check proved `candidate_source` omitted ≡
  `candidate_source=lastfm` bit-for-bit, so no baseline re-run was needed.
- **Statistics:** `eval/lb_paired_bootstrap_ma.py` — paired-difference
  bootstrap keyed on (user, anchor), 5000 resamples, seed 12345, 95%
  percentile CIs. All 348 (user, anchor) pairs are common to both arms.

## Gate verdict

| condition | requirement | observed | pass |
|---|---|---|---|
| (a) reach | pooled CI excludes 0 upward | [+0.0039, +0.0100] | **YES** |
| (b) obscW@k | CI does not exclude 0 downward | [−0.0027, +0.0010] | **YES** |
| (c) pure-discovery | no significant regression | [−0.0546, +0.0172] | **YES** |

**SHIP.** This confirms and strengthens T-C's directional GO: at 6.4× the
sample size the reach gain survives (+9.5% relative) and is consistent
across all three anchors, while the T-C-era worry (MRR/discovery cost)
did not materialize — pooled MRR point estimate is actually *positive*
(+0.0194, n.s.).

## Caveats

- **The win is reach-specific and small in absolute terms** (+0.66pp).
  Reach is the harness's primary discovery metric and the metric the
  mission cares about (artists reached that the user then adopts), but no
  secondary metric independently confirms at significance (recall@k is
  significant in one anchor only).
- **Pure-discovery hits lean negative in the oldest anchor** (2025-06-10:
  −0.0508, CI upper bound exactly +0.0000 — borderline). Pooled it is
  clearly n.s., and the two recent anchors show zero difference, so this
  reads as anchor-age noise (older anchors have thinner LB session data),
  but worth watching post-ship.
- **obscW@k pooled point estimate is −0.0008** — neutral, not positive.
  Blend does not make recommendations *more* obscure; it finds more
  adoptable artists at the same obscurity profile.
- **Latency/cost:** blend adds ListenBrainz + MusicBrainz calls
  (~1 req/s policy-limited) per recommendation build. Fine for the eval
  harness and for the live engine *with caching*; the live-engine
  integration should reuse the same sqlite/TTL cache pattern before the
  lever is flipped in prod.
- The baseline artifact (`vs_mult2.json`) was run in an earlier session;
  arms were not interleaved in time. Both arms are fully deterministic
  given the cached API snapshots, so this is a data-vintage caveat, not a
  randomization one.
